## 二、架构变更：ProviderRegistry 注册表模式

### 2.1 问题

改造前 `MapService::detectPlatform()` 通过字符串匹配识别 Git 平台，兜底 `return 'gitlab'`。URL 不含关键词的自建实例（如 `http://git.company.com/`）被静默误判，且不支持扩展。

### 2.2 方案

新建 `src/Service/Git/ProviderRegistry.php` —— 注册表分离 URL 匹配（轻量 matcher）和 Provider 构造（工厂闭包）：

```
detect(url) → 遍历 matcher → 精确返回平台名 / 抛 ApiException
create(name) → 调用 factory 闭包 → 返回 GitProviderInterface
```

### 2.3 联动修改清单

| 文件 | 联动内容 |
|------|---------|
| `GitProviderInterface.php` | + `getName()` `matchUrl()` `getApiVersion()` 三个方法 |
| `GitlabService.php` | 实现新接口 + Logger 注入 |
| `GiteeService.php` | 同上 |
| `GithubService.php` | 同上 + 分页上限 MAX_PAGES=20 + HTTP header 用日历版本 |
| `GiteaService.php` | **新建** — API v1，`/api/v1/repos/{owner}/{repo}/branches` |
| `GitProviderFactory.php` | 改为 ProviderRegistry 委托（向后兼容封装） |
| `MapService.php` | 注入 Registry；移除 `192.168.137.5:8082` 硬编码；移除 `return 'gitlab'` 兜底 |

## 三、平台检测决策树

```
URL 含 gitlab/gitee/github/gitea 关键词?
  ├─ 是 → 精确匹配, platform_source='auto', detection_method='exact'
  └─ 否 → ProviderRegistry 无匹配
         → 日志 warn "回退默认平台"
         → 使用 DEFAULT_GIT_PLATFORM
         → platform_source='auto', detection_method='fallback'

job_git_map 中手动指定 git_platform?
  └─ 是 → 覆盖自动检测, platform_source='manual', detection_method='manual'
```

**错误信息三级区分：**

| 场景 | 错误输出 |
|------|---------|
| 手动指定 `git_platform: 'gitea'` 但未配 GITEA_BASE_URL | `"Job 'xxx' 在 job_git_map 中指定了 git_platform='gitea'，但该平台尚未配置。请在 .env 中设置 GITEA_BASE_URL..."` |
| 自动检测 fallback，API 调用失败 | `"gitlab 分支查询失败: 404。提示：此平台由 DEFAULT_GIT_PLATFORM 兜底识别...请修改 DEFAULT_GIT_PLATFORM"` |
| 精确匹配，API 调用失败 | 原始错误（不追加兜底提示） |

## 五、新增 Gitea 支持

Gitea 与 GitLab 同为自建平台，在架构中处于同级地位。

**API 对比：**

| | GitLab (自建) | Gitea (自建) | GitHub (SaaS) | Gitee (SaaS) |
|------|------|------|------|------|
| 分支接口 | `/api/v4/projects/{id}/repository/branches` | `/api/v1/repos/{owner}/{repo}/branches` | `/repos/{owner}/{repo}/branches` | `/repos/{owner}/{repo}/branches` |
| 认证头 | `PRIVATE-TOKEN` | `Authorization: token` | `Authorization: token` | `Authorization: token` |
| URL 关键词 | `gitlab` (不可靠) | `gitea` (不可靠) | `github.com` (可靠) | `gitee.com` (可靠) |

**关键认知：自建平台（GitLab/Gitea）的 URL 检测不可靠，必须靠 `default_platform` 或 `job_git_map.git_platform` 兜底。**

### 多实例同平台部署注意事项

**场景：** 需要对接两台 GitLab（或其他同类型平台），一台主实例 + 一台备用/隔离实例。

**核心问题：** `ProviderRegistry` 以平台名称（如 `'gitlab'`）为唯一键，同名注册会静默覆盖。配置层也是扁平结构，`gitlab` 键只能存一个 `base_url`。

**可行方案：** `custom_providers` + `job_git_map` 手动指定，不依赖自动检测。

```php
// 主 gitlab — 正常配置
GITLAB_BASE_URL=http://192.168.1.100
// 第二台 gitlab — custom_providers
'custom_providers' => [[
    'class'  => 'App\\Service\\Git\\GitlabService',
    'config' => [
        'name'     => 'gitlab2',
        'base_url' => 'http://192.168.1.200',
        'token'    => env('GITLAB2_TOKEN', ''),
        'matcher'  => function (string $url): bool {
            return str_contains($url, '192.168.1.200');
        },
    ],
]],
```

**域名命名规范（内网场景）：** 如果内网使用域名而非 IP，**避免域名中包含平台关键词**——如 `gitlab.abc.com` 会被内置 `str_contains($url, 'gitlab')` 匹配器贪婪捕获，导致 custom_providers 的精确匹配器永远不会被轮到。反例和正例：

| ❌ 反例 | ✅ 正例 | 原因 |
|---------|---------|------|
| `gitlab.abc.com` | `code.abc.com` | 不含 gitlab 子串，不会被内置匹配器抢走 |
| `gitea.internal.com` | `git.internal.com` | 同样不含平台关键词 |

> **结论：内网用域名时，命名尽量中性化（`code.` / `git.` / `repo.` / `source.`），不要带 `gitlab`/`gitea`/`github`/`gitee` 字样。用 IP 则完全无此问题。**

## 七、自定义 Git 平台接入

无需修改系统源码。两步完成：

**步骤 1 — 写 Provider 类：**

```php
namespace App\Service\Git;

class BitbucketService implements GitProviderInterface
{
    public function getName(): string { return 'bitbucket'; }
    public function matchUrl(string $url): bool { return str_contains($url, 'bitbucket'); }
    public function getApiVersion(): string { return 'v2'; }
    public function getBranches(string $repository): array { /* API 调用 */ }
}
```

**步骤 2 — 在 settings.php 注册（凭证用 `env()` 从 .env 读取）：**

```php
'custom_providers' => [
    [
        'class'  => 'App\\Service\\Git\\BitbucketService',
        'config' => [
            'name'     => 'bitbucket',
            'base_url' => 'https://api.bitbucket.org/2.0',
            'token'    => env('BITBUCKET_TOKEN', ''),
            'matcher'  => function (string $url): bool {
                return str_contains($url, 'bitbucket');
            },
        ],
    ],
],
```
